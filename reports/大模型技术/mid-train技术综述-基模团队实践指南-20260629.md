# Mid-train 技术综述与落地手册：给 post-train 团队的基模能力塑形指南

作者流程：作者初稿 -> 技术专家挑刺 -> 编辑挑刺 -> 作者修订  
时间范围：近两年公开资料，重点覆盖 2024-06-29 至 2026-06-29  
目标读者：已有 base model / post-train 经验，但尚未系统做过 mid-train 的基模团队  
结论性质：技术研究与工程参考，不构成任何商业或投资建议

## 0. 摘要

Mid-train 不是 post-train 的放大版，也不是简单把 pretrain 再跑一段。本文采用一个保守的工程定义：

> Mid-train 是 base pretrain 后、SFT/RL 等 post-train 前的一段受控继续训练，用来改变模型的能力分布、上下文适应能力，以及后续 SFT/RL 可利用的可训练性。

公开资料中，OLMo 2 明确把 base pretraining 之后、post-training 之前的阶段称为 mid-training；Qwen3、DeepSeek-V3、Llama 3、Qwen2.5-1M 等报告并不总使用这个词，而是称为 pre-training、continued pre-training、context extension 或 long-context pre-training。本文只在工程意义上把这些阶段归入 mid-train-like 范畴，不能等同于原报告术语。

实践上，新团队不应先追大规模训练，而应先做小模型 ablation、数据桶验证、污染检查、base 回归评估，再用主模型短程训练验证方向，并用固定 post-train recipe 检验收益是否保留。Mid-train 的成功标准不是某个 base benchmark 上涨，而是能力增强、遗忘可控、post-train 后收益仍存在。

边界上，本文不把所有 continued pretraining 都等同于 mid-train，也不把 DCA、YaRN、sparse attention、chunked prefill 等推理或架构技巧简单归为训练配方。公开案例只能作为设计参考，不能照搬为 recipe。

对一个只做过 post-train 的团队，最重要的不是立刻上大规模训练，而是先建立四件事：

1. 明确目标能力和不可牺牲能力：例如数学提升不能以通用对话、事实知识、多语能力大幅退化为代价。
2. 做小模型和短程 ablation：先验证数据桶、学习率、序列长度、污染控制和遗忘风险。
3. 建立 mid-train 专用评估：不能只看 pretrain loss，也不能只看 post-train 后的榜单。
4. 让 post-train 团队接力验证：mid-train 的成功标准之一，是 SFT/RL 后收益仍能保留、训练更可控，而不只是 base checkpoint 某几个 zero-shot 分数更好。

### 0.1 读者决策树：你到底该做哪类 mid-train

| 目标 | 优先路线 | 第一轮实验 | 关键评估 | 不建议一开始做 |
| --- | --- | --- | --- | --- |
| 数学、代码、STEM、复杂 QA | Capability mid-train | 小模型数据 mix + 低 LR 短程训练 | 目标能力提升 + 通用能力回撤 | 同时拉长到 128K/1M |
| 32K/128K 长上下文 | Context extension | 逐步长度扩展 + 长文档数据桶 | 多证据、长文档、长代码任务 | 只看 needle-in-a-haystack |
| RL reasoning 上限 | Reasoning substrate mid-train | 验证 base 是否已有 edge-of-competence | 固定 RL/RLVR recipe 接力对比 | 把 RL 当成能力来源本身 |
| 垂直领域能力 | Domain continued pretraining | 低比例领域数据 + 通用保真数据 | 领域任务 + 通用回归 | 用单一领域语料灌满训练 |
| Coding agent / tool use | Code/FIM/trajectory mid-train | FIM、repo-level completion、多文件定位 | repo task、编辑任务、post-train 后 agent eval | 把交互格式全部前移到 mid-train |

这个决策树是执行入口，不是排他分类。一个成熟项目最后可能同时包含 capability shaping、long-context adaptation 和 post-train readiness，但第一轮最好只验证一个主假设。

## 1. 术语边界：什么是 mid-train

### 1.1 一个实用定义

本文采用工程定义，而不是术语洁癖式定义：

> Mid-train 是 base pretrain 后、post-train 前的继续训练阶段。它仍主要使用语言建模目标或其变体，但数据、学习率、序列长度和能力目标已经从“最大覆盖世界文本”转向“高质量能力塑形”。

这一定义覆盖几类公开实践：

- late-stage pretraining / annealing：在较低学习率下，用高质量数据继续训练，追求更低损失和更强下游能力。
- domain / capability continued pretraining：围绕代码、数学、STEM、医学、法律、金融、多语等能力继续训练。
- long-context extension：在原 base context 上扩展到 32K、128K、1M 等，配套位置编码、attention 优化和长文档数据。
- reasoning-oriented mid-train：用数学、代码、合成推理、教材式数据、QA/CoT 等让 base model 更适合后续 RL 或 RLVR。

Mid-train 和 continued pretraining 有重叠，但强调点不同：continued pretraining 描述“训练从哪里接着跑”；mid-train 更强调“为什么跑、用什么数据塑形、如何服务 post-train”。

### 1.2 它和 post-train 的分工

Post-train 团队容易把 mid-train 想成“更多 SFT 数据”或“更早一点的 SFT”。这是危险的，因为两者优化对象不同。

Mid-train 关注：

- 模型内部是否有解决问题所需的知识、算法模式和表征能力。
- 长上下文中位置编码、attention、检索和跨段归纳是否可用。
- 代码、数学、工具轨迹、复杂指令的基础分布是否进入模型参数。
- 后续 SFT/RL 是否更容易把能力激发出来。

Post-train 关注：

- 指令遵循、回答风格、安全拒答、偏好排序。
- 多轮对话和用户意图对齐。
- 推理过程是否外显、是否符合产品策略。
- RL/RLHF/RLAIF/RLVR 中的 reward 可优化行为。

一句话：mid-train 更多改变“会不会”，post-train 更多改变“怎么表现出来”。

### 1.3 不宜把所有训练都叫 mid-train

本文不把以下情况简单归为 mid-train：

- 从零开始的大规模通用预训练。
- 单纯为了对齐、安全或格式的 SFT。
- 只在 post-train 数据上继续跑语言建模，但没有能力目标、数据控制和回归评估。
- RAG、外部工具、推理时 search 等不改变模型参数的系统工程。

边界要清楚，否则团队会用 post-train 的评估和组织方式管理 mid-train，最后很难判断收益来自哪里。

### 1.4 给 post-train 团队的 10 条差异认知

1. Mid-train 的对象是能力分布，不是回答风格。
2. Mid-train 的主损失通常仍接近语言建模目标，不是偏好优化。
3. Mid-train 数据可以包含 QA/CoT/FIM，但不要把 SFT 数据整体前移。
4. Mid-train 的好 checkpoint 不一定是 loss 最低 checkpoint，而是能力提升和遗忘之间的 Pareto 点。
5. Mid-train 的收益必须在 base eval 和 post-train readiness 中同时观察。
6. 数据污染在 mid-train 中比 SFT 更危险，因为它会进入 base 能力判断。
7. 合成数据要先被验证和分层，不应因为“便宜”而提高比例。
8. 长上下文能力要区分训练长度、推理外推长度和线上可服务长度。
9. RL 的收益通常依赖 base 已有可达能力区间，不能把所有 reasoning 问题推给 RL。
10. 放大训练前必须有停止条件；mid-train 不是“预算还没花完就继续跑”。

## 2. 公开证据强弱分层

公开材料不完全一致，最容易出错的地方是把“报告明确写出的训练阶段”“研究者的归纳”和“访谈里的行业经验”混为一谈。本文按证据强弱分三层使用资料。

### 2.1 强证据：技术报告明确披露的阶段设计

强证据指模型团队在技术报告中直接披露的训练阶段、token 规模、上下文长度或数据策略。

| 案例 | 报告原术语 | 和 mid-train 的关系 | 可借鉴点 |
| --- | --- | --- | --- |
| OLMo 2 | mid-training | 狭义 mid-training 样本 | pretraining -> mid-training -> instruction/preference tuning 的清晰分层 |
| Qwen3 | pre-training stages | mid-train-like capability/context stages | 通用训练后增加 STEM/code/reasoning，再做长上下文阶段 |
| DeepSeek-V3 | context extension | mid-train-like long-context extension | 14.8T token 预训练后，分 32K 和 128K 两步扩展 |
| Qwen2.5-1M | long-context training / inference scaling | mid-train-like context adaptation + 推理系统 | progressive length extension 与 DCA/YaRN/部署优化分工 |
| Phi-4 | pretraining / midtraining / post-training | synthetic-heavy training strategy | 合成数据贯穿训练，但显式 midtraining 主要承担 4K -> 16K context |
| Llama 3 | continued pre-training / annealing | mid-train-like late-stage training | 长上下文继续训练和末期 annealing 的保守样本 |

这些报告能支撑“阶段设计”的存在，但通常不能支撑“照搬数据比例或学习率”。未公开的 tokenizer、数据清洗、采样策略、基础 checkpoint 状态、优化器状态和系统实现，都会显著影响结果。

### 2.2 中证据：综述和受控实验的归纳

Mid-training survey、PRISM 以及 pre-training / mid-training / RL interplay 这类研究能帮助团队理解变量关系：数据保留与能力迁移、能力是否位于模型可达边界、RL 是否能从 mid-trained checkpoint 获益等。

这些研究的价值是提出机制假设和实验维度，而不是给出可直接外推的产业配方。尤其是受控 synthetic task 或小规模模型上的结论，不能直接升级成“所有大模型训练定律”。

### 2.3 弱证据：访谈、博客和行业口径

访谈和博客通常能提供两类信息：术语如何被从业者使用，以及团队如何描述训练阶段的组织逻辑。但这类资料常常省略失败实验、数据细节和工程约束，所以只能作为趋势背景，不能作为硬事实。

如果访谈声称某个阶段“非常重要”，本文会把它当作待验证假设；只有技术报告、可复现实验或团队内部 ablation 支持后，才进入训练决策。

### 2.4 从公开资料抽出的保守共识

第一，mid-train-like 阶段通常发生在 base 能力形成之后、post-training 之前。OLMo 2 是狭义样本；Qwen3、DeepSeek-V3、Qwen2.5-1M、Llama 3 是工程上相近但术语不同的样本。

第二，在预训练后期，公开案例普遍提高高质量数据占比；但收益仍由模型规模、token budget、数据多样性、学习率、上下文长度和评估目标共同决定。更稳妥的说法是：晚期 token 的边际价值更依赖质量和配比，而不是简单追加 web tokens。

第三，长上下文不是只改 RoPE scaling。真实长上下文能力需要训练数据、长度 curriculum、位置/注意力机制、推理框架和评估任务共同成立。DCA、YaRN、sparse attention、chunked prefill 等技术要区分其作用位置：有些影响训练可行性，有些主要服务推理外推和部署成本。

第四，reasoning mid-train 更适合作为“提高 RL 可用 substrate”的强工程假设，而不是已被证明的普遍定律。较保守的说法是：如果 base/mid-trained checkpoint 对目标任务完全没有可解性，RL 通常很难稳定地产生能力；但 RL 是否带来真实能力增益，还取决于 pretraining 暴露、任务是否位于模型能力边界、reward 设计和探索预算。

第五，合成数据有价值，但不是免费午餐。Phi-4 说明 synthetic data 可以显著影响 reasoning 表现，也提醒团队仍需要有机数据来补世界知识、表达多样性和真实分布。

第六，透明的实验台账比单点榜单更重要。OLMo 2 这类报告的最大价值之一，是把 pretraining、mid-training、post-training 和数据构成分开描述，帮助后来团队建立自己的 lineage、分桶 loss 和回归评估。

## 3. 代表案例拆解

### 3.1 OLMo 2：最清楚的 mid-training 公开范式

OLMo 2 明确把第二阶段称为 mid-training：第一阶段是大规模预训练，第二阶段使用高质量数据混合进行训练，然后再做 instruction tuning 和 preference tuning。这个分层非常适合作为新团队的组织模板。

可复用点：

- 将 mid-train 独立成一个有输入输出、有 eval gate 的阶段。
- 不把所有数据混在一起，而是明确区分 Dolma、DCLM、StarCoder、OpenWebMath 等来源。
- 在 post-train 前保留 base checkpoint 的独立评估。
- 使用 micro-annealing 等小规模后期训练实验，帮助判断是否值得把配方放大。

需要注意：

- OLMo 2 的公开配方不等于所有团队的最优配方。它的价值是透明和可学习，不是可直接复制。
- 其数据结构偏开源英文生态，中文、多语、工具使用、agentic coding 等目标需要重做数据桶。

### 3.2 Qwen3：通用 -> 推理/代码/STEM -> 长上下文

Qwen3 技术报告披露了三阶段预训练路线：

- 第一阶段：总预训练约 36T tokens，其中 S1 超过 30T tokens，覆盖通用语言知识。
- 第二阶段：约 5T 高质量 STEM、代码、推理和合成数据，增强任务相关能力。
- 第三阶段：数千亿长上下文 tokens，把训练长度扩展到 32K。

这说明一个成熟团队会把 late-stage training 设计成多个能力目标，而不是一个单一“继续训练”阶段。

可复用点：

- 把能力型 mid-train 和长上下文 mid-train 分开设计。
- 第二阶段更像 capability shaping，第三阶段更像 context adaptation。
- 区分训练长度和推理支持长度：Qwen3 报告中的训练阶段扩到 32K，部分发布模型再通过 ABF、YaRN、DCA 等位置/推理技术支持更长上下文。
- 评估也应分开：STEM/代码/推理看专项 benchmark 和真实任务；长上下文看长文档、长代码和多证据任务。

需要注意：

- Qwen3 的完整数据配比没有完全公开，不能把其宏观阶段当成可复制 recipe。
- 如果团队资源有限，建议先做第二阶段能力增强，再决定是否做极长上下文。

### 3.3 DeepSeek-V3：大规模 base 后的 context extension 和工程约束

DeepSeek-V3 报告给出了两个对 mid-train 很重要的信号：

- 预训练阶段使用 14.8T 高质量、多样 token。
- 长上下文扩展采用两阶段策略：先扩到 32K，再扩到 128K。

此外，DeepSeek-V3 的训练还包含 MLA、DeepSeekMoE、auxiliary-loss-free load balancing、MTP 等架构和训练设计。对 mid-train 团队来说，这提醒我们：如果 base 架构、位置编码、MoE 路由、attention memory 和推理框架不支持，后期数据再好也可能达不到目标。

可复用点：

- 长上下文扩展应分阶段，不宜一步拉到目标长度。
- Context extension 前要确认推理框架和 attention kernel 可承载训练后的部署形态。
- 多目标训练时要监控 MoE load balance、训练稳定性和各数据桶 loss。

需要注意：

- DeepSeek-V3 是从架构到训练都深度协同的大系统，不能把长上下文阶段孤立出来照搬。
- 其报告中很多关键效率来自系统和架构，数据/训练阶段只是整体的一部分。

### 3.4 Phi-4：synthetic-heavy training strategy，而不是干净的 reasoning mid-train

Phi-4 更适合作为 synthetic-heavy training strategy 的案例，而不是一个干净的“reasoning mid-train”配方。报告确实强调 synthetic data for pretraining and midtraining，但其显式 midtraining stage 主要承担长上下文扩展，把 context 从 4K 扩到 16K；reasoning 增益来自合成数据、课程设计、数据混合和 post-training 的共同结果。

可复用点：

- 数学、代码、逻辑推理、教材式解释非常适合引入可验证或半可验证合成数据。
- 合成数据最好有生成、过滤、验证、去重、难度分层和风格多样化流程。
- 小模型上先看合成数据占比、题型难度、解答长度、正确性过滤对最终能力的影响。

需要注意：

- 合成数据会放大 teacher model 的偏见和错误。
- 如果生成模板单一，模型会学到“看起来像推理”的表面格式。
- 对开放域知识、真实用户 query 和多语言表达，仍需混入高质量有机数据。

### 3.5 Qwen2.5-1M：训练扩展、推理外推和部署优化要分开

Qwen2.5-1M 报告展示了长上下文能力如何由训练、推理外推和部署系统共同构成。需要特别避免一个误读：1M context 不是简单通过 mid-train 数据直接训出来的。

更准确的拆法是：

- Long-context pre-training：采用 progressive pre-training，大致按 4K -> 32K -> 64K -> 128K -> 256K 的长度路线推进，并使用自然长文档、合成长上下文任务和长度分布 curriculum。
- Post-training：用长指令数据和多阶段 post-training 让模型更会在长上下文中完成用户任务。
- Inference / deployment：1M 支持主要依赖 DCA、YaRN 等 length extrapolation 与推理框架优化；稀疏注意力、chunked prefill、vLLM/BladeLLM 适配主要服务部署成本，而不是普通意义上的 mid-train 数据配方。

可复用点：

- 长上下文是模型、数据、训练、推理框架四件事共同定义的能力。
- 从 4K 直接到 1M 风险极高，建议先按 8K/16K/32K/128K/256K 逐步验证训练与推理的一致性。
- 评估要避免单一 needle 指标，应包含复杂长文档 QA、长代码、表格、跨段推理和鲁棒性测试。

需要注意：

- 1M context 的成本和部署复杂度非常高，不一定适合所有产品。
- 对多数团队，32K/128K 高质量可用性可能比 1M 展示能力更有商业价值。

### 3.6 Llama 3：continued pre-training + long-context + annealing 的保守样本

Llama 3 系列报告没有把相关阶段称为 mid-training，但它给了一个很有价值的保守样本：先完成大规模通用预训练，再通过 continued pre-training 扩展长上下文，并在末期进行学习率 annealing 和高质量数据 mix 调整。

可复用点：

- late-stage training 可以同时承担长上下文适应和质量收敛，但必须持续做通用能力回归。
- annealing 不是“再训一点”的口号，而是和数据质量、学习率、checkpoint 选择绑定的阶段。
- 如果团队保守，Llama 3 的启示不是追求复杂花活，而是把继续训练、长上下文和末期收敛拆成可观测步骤。

需要注意：

- Llama 3 报告里的具体 token 数和数据配方依赖其大规模基础训练，不宜迁移成小团队默认预算。
- 其长上下文和后续指令模型能力也受到 post-training 和系统实现影响。

### 3.7 PRISM 与 pre/mid/RL 研究：把机制假设当假设

PRISM 和 pre-training / mid-training / RL interplay 这类论文的价值，是把几个长期靠经验讨论的问题变成可实验的问题：mid-training 如何影响 retention，哪些能力在继续训练中相互促进或相互干扰，RL 是否依赖 pre/mid 阶段已经暴露过的能力边界。

对团队的可复用点：

- 把“保留旧能力”和“注入新能力”拆成两个指标，不要只看目标能力增长。
- 把任务是否位于模型 edge-of-competence 作为 RL 前置检查。
- 把 controlled experiment 的结论作为 ablation 设计参考，而不是直接作为主训练结论。

需要注意：

- 这类论文通常规模、任务和数据分布都受控，外推到产业模型时必须重新验证。
- 对 reasoning 的判断尤其要保守：RL 可能放大已有能力，也可能放大奖励漏洞和格式偏差。

## 4. Mid-train 的设计框架

一个可执行的 mid-train 项目，应先回答五个问题：

1. 目标能力是什么？
2. 哪些能力不能退化？
3. 这些能力分别由哪些数据桶驱动？
4. 训练长度、学习率和 context curriculum 如何设置？
5. 如何证明它对 post-train 有帮助？

### 4.1 能力目标分层

建议把目标分成四层：

第一层：通用 base 保真

- 通用语言建模 loss 不异常上升。
- 常识、事实、多语、基础指令理解不明显退化。
- 多轮对话虽然不是 mid-train 主目标，但基本可读性不能崩。

第二层：核心能力增强

- 数学：GSM8K、MATH、AIME 类，以及内部可验证题库。
- 代码：HumanEval、MBPP、EvalPlus、repo-level completion、FIM、多文件定位等；SWE-bench 更适合在固定 harness、轻量 instruction wrapping 或 post-train readiness 阶段使用。
- STEM：GPQA、MMLU-Pro、领域题库。
- 推理：BBH、ARC、逻辑推理、复杂 QA。

第三层：结构能力

- 长上下文检索、综合、冲突消解。
- FIM、代码补全、多文件编辑。
- 表格、日志、长 PDF、长会议纪要理解。

第四层：post-train readiness

- SFT 后是否更容易学会目标格式。
- RL/RLVR 后是否更稳定提升。
- reward hacking 是否减少。
- 推理轨迹是否更可控。

### 4.2 数据桶设计

建议至少建立以下数据桶：

| 数据桶 | 目标 | 主要风险 | 建议控制 |
| --- | --- | --- | --- |
| 高质量 web / book | 保持通用语言与世界知识 | 噪声、重复、版权/合规 | 去重、质量分类、采样上限 |
| 代码 | 编程能力、形式化结构 | license、重复、benchmark 泄漏 | license 过滤、repo 去重、题库隔离 |
| 数学 / STEM | 推理与符号能力 | 过拟合竞赛题、解答错误 | 可验证题、难度分层、proof/answer 校验 |
| 合成教材 | 概念压缩、教学式推理 | 风格单一、teacher 错误 | 多 teacher、多模板、验证器过滤 |
| QA / CoT | 推理轨迹和任务格式 | 把 post-train 行为提前固化 | 控制比例，保留自然文本 |
| 长文档 | 长上下文适应 | 空洞 padding、低效 token | 真实长文档 + 合成长任务 |
| FIM / 代码编辑 | coding agent 能力 | 破坏普通 LM 分布 | 分桶 loss 和代码 eval |
| 领域数据 | 行业/产品能力 | 灾难性遗忘、狭窄化 | 低比例混入，通用回归 |

不要只做一个全局 mix。每个桶要有：

- 数据来源和 license 记录。
- 去重策略。
- 质量打分。
- contamination 检查。
- 采样比例。
- 单独 loss。
- 对应 eval。

### 4.3 学习率和 token budget

公开报告通常不会完整给出所有细节，但工程上有几条稳健原则。

第一，mid-train 通常不应使用接近 pretrain 初期的高学习率。模型已经形成大量通用能力，过高学习率会导致遗忘、风格漂移和 loss spike。

第二，学习率策略要和目标匹配：

- 短程能力增强：低学习率继续训练，观察专项能力和通用回归。
- late-stage annealing：进一步降低学习率，用高质量数据收敛。
- context extension：学习率更保守，并配合长度 curriculum。
- domain adaptation：更要限制 token budget 和数据比例，避免模型变窄。

第三，token budget 不是越大越好。下面三档只是启动实验的保守刻度，不按模型规模线性通用，也不能替代 scaling ablation：

- 1B-10B tokens：用于 pipeline、数据质量、LR、污染检查和 proxy ablation。
- 10B-100B tokens：用于可观察的能力增强和中小模型 recipe 验证。
- 100B-1T+ tokens：才适合主模型较完整的 capability shaping 或 long-context extension。

以上不是行业定律，而是工程上较保守的起步刻度。真正 token budget 应由模型规模、数据质量、目标能力和预算共同决定。

### 4.4 第一轮 ablation 设计表

第一轮不要追求“最好配方”，而要快速排除错误方向。建议至少跑以下 6 类实验：

| 实验目的 | 模型规模 | Token budget | 数据 mix 变量 | LR 变量 | 成功指标 | 停止条件 |
| --- | --- | --- | --- | --- | --- | --- |
| Pipeline 和 eval 校准 | proxy model | 1B-3B | 小样本全桶 mix | 低 LR 单点 | loss 正常、eval 可复现 | loss spike、eval 不稳定 |
| 通用保真底线 | proxy model | 3B-10B | 通用高质量占比 30/50/70% | 固定低 LR | 通用能力回撤可控 | 多语/事实大跌 |
| 数学/STEM 注入 | proxy model | 5B-20B | 数学/STEM 10/20/30% | 低 LR + 更低 LR | 数学提升且通用不崩 | 只提升公开题型 |
| 代码/FIM 注入 | proxy model | 5B-20B | 代码/FIM 10/20/30% | 固定低 LR | completion/FIM/repo task 提升 | 普通生成明显代码化 |
| 合成数据占比 | proxy model | 5B-20B | synthetic 5/15/30% | 固定低 LR | 可验证任务提升、风格多样 | 模板化或污染风险 |
| 主模型短程验证 | target model | 10B-100B | 候选 2-3 个 mix | 低 LR sweep | Pareto checkpoint 出现 | 目标能力不涨或遗忘过大 |

这些实验的产出不是最终模型，而是决策：哪个数据桶有效，哪个比例危险，哪个学习率区间安全，是否值得进入更长主训练。

### 4.5 Curriculum：先能力，后长度，还是并行？

建议新团队不要一开始把所有目标混在一起。

更稳的顺序是：

1. Base 保真检查：确认 checkpoint、tokenizer、optimizer state、数据 pipeline、评估体系可用。
2. 能力型 mid-train：数学/代码/STEM/高质量合成数据，短程 ablation。
3. 长上下文适应：从较短 context 逐步拉长，单独监控长上下文 loss 和真实任务。
4. 小规模 post-train 接力：用固定 SFT/RL recipe 比较 mid-train 前后 checkpoint。
5. 决定是否合并为主训练配方。

为什么建议先能力后长度？因为长上下文训练成本高、debug 难，而且容易把架构和数据问题混在一起。如果模型基础 reasoning/代码能力没有改善，长上下文只是放大输入窗口，不一定放大任务能力。

也有例外：如果产品目标就是法律/金融长文档 QA 或代码库 agent，长上下文可以前置，但仍建议先在小模型上验证。

## 5. 训练工程检查点

### 5.1 启动前检查

启动 mid-train 前，应完成：

- Base checkpoint 的完整 eval 快照。
- Optimizer state 是否接续的决策。
- Tokenizer 和 special tokens 冻结。
- 数据去重和 benchmark contamination 检查。
- 每个数据桶的样例审查。
- 小规模 dry run，确认 loss、吞吐、显存、checkpoint、resume 正常。
- Post-train 团队确认后续 SFT/RL 的固定 recipe。

如果没有 post-train 接力验证，mid-train 很容易优化错目标。

### 5.2 训练中监控

至少需要监控：

- 全局 training loss / validation loss。
- 按数据桶分解的 validation loss。
- 通用能力 benchmark 的回归。
- 目标能力 benchmark 的增益。
- 长上下文不同长度段的 loss。
- 生成样例的风格漂移。
- MoE 模型的专家负载、路由熵和 token drop。
- 梯度范数、loss spike、checkpoint 间差异。

不要等训练结束才做 eval。Mid-train 的失败常常在早期就有信号：通用 eval 大跌、输出风格模板化、某个数据桶 loss 异常、长上下文短任务退化。

### 5.3 Checkpoint 策略

建议保留：

- pre-mid checkpoint。
- 每个阶段的 early/mid/final checkpoint。
- 学习率变化前后的 checkpoint。
- context length curriculum 每个长度阶段的 checkpoint。
- 用于 post-train 接力的候选 checkpoint。

不要只保留最后一个。Mid-train 的最佳 checkpoint 可能不是 loss 最低的 checkpoint，而是目标能力和遗忘之间的 Pareto 最优点。

### 5.4 容易低估的工程风险

第一，mixed-length batching 和 context parallelism 会带来负载不均。长上下文 batch 的吞吐、显存碎片、padding 浪费和通信模式都和短上下文不同，训练监控必须按长度段拆开，否则全局吞吐会掩盖局部不稳定。

第二，optimizer state 接续或重置都可能出问题。接续 optimizer state 更像原训练轨迹延长，但可能继承旧阶段动量；重置 optimizer state 更干净，但需要重新设计 warmup，否则容易出现 loss spike 或过慢收敛。两种方案都应先在 proxy scale 测。

第三，训练 attention kernel 和部署 attention kernel 不一致会制造假阳性。离线 eval 通过，不代表线上 128K/1M 服务可用；长上下文训练、推理外推、kv cache、chunked prefill、sparse attention 和 serving 框架必须联合验证。

## 6. 评估体系

### 6.1 Base 层评估

Base 层评估应覆盖：

- 通用知识：MMLU、MMLU-Pro、内部知识题。
- 推理：BBH、ARC、GPQA、逻辑题。
- 数学：GSM8K、MATH、AIME 风格内部题。
- 代码：HumanEval、MBPP、EvalPlus、repo-level completion、FIM、多文件定位；SWE-bench 更适合放在固定 harness 或 post-train readiness 中。
- 多语：中文、英文和目标市场语言。
- 生成质量：长回答、摘要、解释、翻译、自然对话。

注意：post-train 后榜单提升不能反推 mid-train 一定成功。必须保留 base checkpoint 的独立对比。

### 6.2 长上下文评估

长上下文评估要分三层：

第一层：机械检索

- Needle-in-a-haystack。
- 多 needle。
- 不同位置、不同长度、不同噪声密度。

第二层：真实理解

- 长文档 QA。
- 多章节摘要。
- 多证据综合。
- 冲突信息判断。
- 长表格/日志分析。

第三层：任务完成

- 长代码库修改。
- 多文件 bug 定位。
- 法律/财报/论文综合分析。
- Agent 轨迹复盘和下一步决策。

如果只测第一层，模型可能学会“找字符串”，但不具备真实长上下文工作能力。

### 6.3 Post-train readiness 评估

给同一套 SFT/RL recipe，比较：

- pre-mid base -> post-train 后表现。
- mid-trained base -> post-train 后表现。

重点看：

- 同样 SFT 数据下是否更快收敛。
- RL 是否更稳定，是否更少 reward hacking。
- 推理任务是否更能从 RLVR 受益。
- 安全和拒答是否更难对齐。
- 输出风格是否被 mid-train 数据提前污染。

这一步对 post-train 团队特别重要：mid-train 的收益如果不能在 post-train 后保留，就不应算作真实收益。

## 7. 常见失败模式

### 7.1 把 benchmark 当训练目标

表现：内部指标飞涨，真实任务无改善。

原因：

- 数据中混入 benchmark 或近似题。
- 合成数据围绕公开题型生成。
- 评估集太窄。

处理：

- 使用私有 holdout。
- 做 n-gram / embedding / 题型相似度污染检查。
- 用真实任务集和人工审查补充。

### 7.2 合成数据导致风格坍缩

表现：模型回答都像同一个 teacher，推理步骤模板化，遇到非标准题反而脆弱。

处理：

- 多 teacher、多 prompt、多解法。
- 混入有机数据。
- 控制 CoT 数据比例。
- 用短答、长答、无 CoT、有 CoT 多种格式混训。

### 7.3 长上下文“指标好看、任务难用”

表现：needle 近满分，但长文档综合、长代码任务差。

处理：

- 增加真实长任务。
- 构造多证据和冲突证据测试。
- 测不同长度、不同信息密度。
- 对短上下文能力做回归，避免长训伤短任务。

### 7.4 灾难性遗忘和能力偏科

表现：数学/代码涨了，通用问答、多语、写作明显下降。

处理：

- 保留通用高质量数据桶。
- 降低专项数据占比。
- 缩短训练或降低学习率。
- 从更早 checkpoint 重新做 mix ablation。

### 7.5 把 post-train 问题前移错了

表现：mid-train 数据里塞大量指令、偏好、安全、格式数据，base 模型行为过早固化。

处理：

- 把“能力数据”和“行为数据”分开。
- Mid-train 可以少量引入 QA/CoT，但不要替代 SFT。
- 用 base eval 和 post-train readiness 同时判断。

## 8. 给新团队的 90 天路线图

### 第 0-14 天：定义目标和评估

交付物：

- Mid-train charter：目标能力、不可牺牲能力、目标 checkpoint、预算。
- Eval matrix：base eval、能力 eval、长上下文 eval、post-train readiness。
- 数据桶设计文档。
- 风险清单和 go/no-go gate。

Go/no-go：

- 如果没有稳定 eval 和污染检查，不进入训练。
- 如果 post-train 团队没有固定接力 recipe，不进入主实验。

### 第 15-35 天：数据和小模型 ablation

交付物：

- 1B/3B/7B proxy model 上的数据 mix 实验。
- 学习率和 token budget sweep。
- 合成数据正确性和多样性报告。
- 每个数据桶的 loss 和能力关联分析。

Go/no-go：

- 至少一个 mix 在目标能力上提升，且通用能力回撤可接受。
- 合成数据没有明显污染和风格坍缩。

### 第 36-60 天：主模型短程 mid-train

交付物：

- 主模型 10B-100B token 级短程训练。
- 每日 checkpoint eval。
- 能力/遗忘 Pareto 曲线。
- 候选 checkpoint 选择。

Go/no-go：

- 目标能力提升不是单一 benchmark 驱动。
- 通用能力下降低于事先定义阈值。
- 生成样例没有明显风格漂移。

### 第 61-75 天：post-train 接力验证

交付物：

- 固定 SFT recipe 对比。
- 如适用，固定 RL/RLVR recipe 对比。
- post-train 后真实任务评估。
- reward hacking / safety / style 回归报告。

Go/no-go：

- Mid-trained checkpoint 在 post-train 后仍保留收益。
- 对齐难度没有显著增加。

### 第 76-90 天：放大或收缩决策

交付物：

- 最终 recipe。
- 是否进入 100B-1T+ token 训练的决策。
- 数据和评估自动化改进清单。
- 长上下文或第二能力目标的下一阶段计划。

Go/no-go：

- 如果收益主要来自污染、窄评估或 post-train 偶然性，不放大。
- 如果收益稳定、可解释、可在 post-train 后保留，进入主训练。

## 9. 可执行 checklist

### 9.1 数据 checklist

- [ ] 每个数据源有 license / 合规记录。
- [ ] 每个数据桶有质量打分和人工样例审查。
- [ ] benchmark contamination 检查完成。
- [ ] 训练集、验证集、私有 holdout 分离。
- [ ] 合成数据有正确性过滤。
- [ ] 合成数据有多样性和模板化检查。
- [ ] 长上下文数据包含真实长任务，不只是拼接文本。
- [ ] 通用数据保留比例有 ablation 支持。

### 9.2 训练 checklist

- [ ] Base checkpoint eval 完成。
- [ ] Tokenizer / special tokens 冻结或变更有充分理由。
- [ ] Optimizer state 接续策略明确。
- [ ] LR schedule 和 warmup/decay 记录清楚。
- [ ] 每个阶段有 checkpoint 保存策略。
- [ ] 分桶 validation loss 可观测。
- [ ] MoE/attention/long-context 系统指标可观测。
- [ ] 训练可 resume，checkpoint 可复现。

### 9.3 评估 checklist

- [ ] Base eval、能力 eval、长上下文 eval 分开。
- [ ] 指标包含真实任务和人工审查。
- [ ] 不只看平均分，也看能力回撤。
- [ ] 每个候选 checkpoint 做同一套 eval。
- [ ] Post-train 接力 recipe 固定。
- [ ] Post-train 后收益可复现。

### 9.4 组织 checklist

- [ ] 数据团队、训练团队、post-train 团队共用同一张实验台账。
- [ ] 每次实验记录数据 mix、LR、token budget、context length、eval 版本。
- [ ] 训练中有停止条件。
- [ ] 放大训练前必须过 go/no-go gate。
- [ ] 发布前保留 pre-mid、mid、post 三类 checkpoint 的完整报告。

## 10. 对只做过 post-train 团队的特别提醒

第一，不要用 SFT 的心智做 mid-train。Mid-train 数据可以包含 QA、CoT、代码任务，但其目标不是让模型学会产品回答格式，而是改变能力分布。

第二，不要让 RL 背所有锅。如果 base model 没有足够的数学、代码、长上下文和自我纠错 substrate，RL 很可能只学到表面搜索或冗长推理。

第三，不要只追榜。Mid-train 的真实收益要在私有任务、真实产品任务和 post-train 接力中同时成立。

第四，不要一开始就追 1M context。对多数团队，先把 32K/128K 做到可靠，比展示极长 context 更实际。

第五，不要忽视数据工程。Mid-train 项目中，训练代码常常不是瓶颈，数据 lineage、污染控制、分桶 eval 和实验纪律才是。

## 11. 第一轮搜索空间

如果团队已有一个可用 base model 和稳定 post-train pipeline，建议第一轮不要做大而全，先做一个“能力型 mid-train + post-train 接力”的最小闭环。以下比例不是推荐 recipe，而是用于启动 ablation 的搜索空间；不同模型规模、预训练历史、目标语言和产品任务会显著改变最优 mix，主训练前必须通过 proxy scale 和短程主模型验证。

目标：

- 提高数学、代码、STEM 和复杂 QA。
- 不显著伤害通用、多语、事实和自然生成。
- 验证 SFT/RL 后是否仍有收益。

数据 mix 搜索空间：

- 40%-50% 高质量通用文本。
- 15%-25% 代码。
- 15%-25% 数学/STEM。
- 5%-15% 合成教材/可验证推理。
- 5%-10% QA/CoT/FIM。

这些区间只用于第一轮搜索，不是结论。实际比例必须通过小模型和短程主模型实验决定。

训练起点：

- 低学习率继续训练。
- 先用原 context length 或小幅扩展。
- 10B-50B tokens 级短程主模型实验；如果模型较小或数据极窄，预算还应继续下调。
- 每 10%-20% token 保存 checkpoint 并 eval。

评估起点：

- Base eval：MMLU/MMLU-Pro、BBH、GPQA、GSM8K/MATH、HumanEval/MBPP、多语集。
- 私有 eval：内部数学、代码、复杂 QA、真实用户任务。
- Post-train readiness：固定 SFT recipe，对比 pre-mid 和 mid checkpoint。

如果这轮闭环没有稳定收益，不要急着扩大 token budget。先回到数据和学习率。
