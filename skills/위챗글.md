# 위챗 공식계정 아티클: 작가-편집자-독자 3-에이전트 협업

$ARGUMENTS 에 대해 심층 리서치를 수행하고, 바로 발행 가능한 위챗(WeChat) 공식계정 아티클을 산출하라. 세 에이전트가 각자의 역할을 수행한다: 작가는 심층 초고를 쓰고, 편집자는 구조와 표현을 다듬으며, 독자는 타깃 독자층의 관점에서 검토한다.

**지원하는 입력 형식**: 주제 설명, 예: `대형 언어모델 OPD 기술 해설`, `Qwen3 기술보고서 해설`, `버핏은 왜 기술주를 사지 않는가`

---

## 설계 철학

좋은 공식계정 아티클은 다음 세 가지 차원을 동시에 만족해야 한다:
1. **깊이** — 시간을 들여 읽을 만한 가치가 있어야 한다(작가 담당)
2. **가독성** — 구조가 명확하고 리듬이 좋으며 중도 이탈을 유발하지 않아야 한다(편집자 담당)
3. **실제로 이해가 되어야 함** — 타깃 독자가 중간에 포기하지 않아야 한다(독자 담당)

혼자 쓰면 "자기 만족"에 빠지기 쉽다 — 쓴 사람은 명확하다고 느끼지만 읽는 사람은 이해하지 못한다. 3-에이전트 협업의 본질은 **외부 관점을 강제로 도입하는 것**이다.

---

## 1단계: 리서치와 소재 수집

### 1-1: 아티클 포지셔닝 명확화

작성을 시작하기 전에 다음 정보를 먼저 확인한다(사용자가 지정하지 않았다면 먼저 물어본다):

| 차원 | 확인해야 할 것 | 기본값 |
|------|---------|--------|
| **타깃 독자** | 기술적 배경 수준 | 어느 정도 기술 배경은 있지만 해당 분야 전문가는 아님 |
| **아티클 깊이** | 대중 교양/중간 깊이/하드코어 | 중간 깊이(수식은 있지만 명확히 설명함) |
| **아티클 분량** | 글자 수 범위 | 3000~4000자 |
| **원본 논문/자료 다운로드 필요 여부** | PDF/삽화 필요 여부 | 필요함 |
| **작문 스타일** | 격식체/대화체/날카로운 톤 | 대화체(똑똑한 친구에게 설명하듯이) |

### 1-2: 심층 리서치

Agent 도구를 사용해 **병렬로** 2~3개의 리서치 에이전트를 실행하여 충분한 소재를 수집한다:

**리서치 에이전트 A: 핵심 내용 리서치**
- 논문 해설인 경우: 논문 PDF를 다운로드하고, 핵심 기여도, 주요 도표, 실험 결과를 추출
- 기술 주제인 경우: 최신 동향, 핵심 논문, 기술 세부사항을 검색
- 비즈니스/투자 주제인 경우: 최신 데이터, 산업 리포트, 경쟁 구도를 검색

**리서치 에이전트 B: 산업 배경과 응용**
- 해당 기술/주제의 산업 적용 현황을 검색
- 어떤 회사들이 사용하고 있는가? 효과는 어떤가?
- 최신 발전 추세와 이정표가 되는 사건들

**리서치 에이전트 C (선택): 경쟁/비교 리서치**
- 유사한 방법/제품과의 비교
- 역사적 발전 과정
- 향후 진화 방향

### 1-3: 소재 프레임 정리

모든 리서치 에이전트가 완료되면 다음을 정리한다:
1. **핵심 논지**(아티클이 전달하고자 하는 핵심 메시지를 한 문장으로 요약)
2. **핵심 데이터**(가장 임팩트 있는 데이터 포인트 3~5개)
3. **삽화 목록**(어떤 그림이 필요하고, 출처는 무엇인지)
4. **아티클 개요**(6~8개 챕터의 제목과 핵심 내용)

## 2단계: 작가 에이전트가 초고 작성

Agent 도구를 사용해 **작가 에이전트**를 실행하고, 상세한 작문 지시를 전달한다.

### 작가 에이전트 프롬프트 템플릿

```
당신은 심층 기술 작문 전문가(Author Agent)이며, 위챗 공식계정 아티클을 작성해야 한다.

## 타깃 독자
{1단계에서 확인한 독자 프로필}

## 작문 스타일 요구사항
- 순수 한국어 표현을 사용하고, 불필요한 외국어 혼용을 피한다(전문 용어는 처음 등장할 때 영어 원어를 병기하고, 이후에는 한국어로 표기)
- 학술 논문의 번역이 아니라, 똑똑한 친구에게 설명하는 기술 교양 콘텐츠처럼 쓴다
- 이해를 돕는 비유를 사용하되, 비유는 적절하고 진부하지 않아야 한다
- 핵심 수식/데이터는 반드시 포함하되, 각각을 쉬운 말로 풀어서 설명한다
- 이모지를 사용하지 않는다
- 문단은 4줄을 넘기지 않는다(공식계정의 열람 환경을 고려)

## 핵심 내용
{정리된 소재, 데이터, 논지}

## 아티클 구조 요구사항
1. **도입부(첫 3문단)**: 반드시 강력한 훅이 있어야 한다 — 데이터의 임팩트나 직관에 반하는 결론으로 시작하고, 온건한 비유로 시작하지 않는다
2. **배경**: 왜 이것이 중요한가? 어떤 문제를 해결하는가?
3. **핵심 내용(2~3절)**: 기술적 깊이는 여기서 드러나되, 각 기술 포인트마다 "쉬운 말 번역"이 있어야 한다
4. **실증/사례**: 데이터와 사례로 말하고, 공허한 주장을 하지 않는다
5. **산업에 대한 영향/전망**: 이 일이 산업에 어떤 의미인지
6. **마무리**: 전파력 있는 한 문장의 판단으로 마무리하며, 캡처해서 공유하기 좋아야 한다

## 삽화 요구사항
- 논문 해설류 아티클: 반드시 논문 PDF에서 원본 그림을 추출하여 `![설명](상대경로)` 형식으로 아티클에 직접 삽입한다. `[그림X: 설명]` 같은 자리표시자를 사용하지 않는다
- 추출 방법: pdftoppm으로 PDF 페이지를 고해상도 PNG로 렌더링(최소 900 DPI)한 후, PIL로 원하는 도표 영역을 잘라낸다
- 각 그림은 500KB 이상으로, 고화질을 보장한다
- 이미지는 `assets/{주제 약칭}/` 디렉토리에 통일하여 저장한다
- 논문류가 아닌 아티클: 삽화가 필요하면 적절한 이미지를 검색하여 다운로드하고, 마찬가지로 직접 삽입한다

## 수식 요구사항
- 모든 수학 공식은 LaTeX 형식을 사용한다: 인라인은 `$...$`, 독립 수식은 `$$...$$`
- 순수 텍스트로 수식을 작성하는 것(예: `> D_KL(P || Q) = ...`)을 금지하며, 반드시 LaTeX 렌더링 형식을 사용해야 한다
- 각 수식 뒤에는 반드시 "쉬운 말로 번역하면……" 형태의 설명을 붙인다

완성된 아티클 초고를 작성하라. 목표 분량은 약 {목표 글자 수}자다.
```

### 작가 에이전트 완료 후

초고 파일이 생성되었는지 확인하고, 전문을 읽어 내용의 완결성을 확인한다.

## 3단계: 편집자 에이전트 + 독자 에이전트 병렬 검토

초고가 완성되면 Agent 도구를 사용해 **같은 메시지 안에서** 편집자 에이전트와 독자 에이전트를 실행한다.

### 편집자 에이전트 프롬프트 템플릿

```
당신은 베테랑 공식계정 편집자(Editor Agent)다. 다음 아티클을 정밀하게 검토하라.

## 검토 기준
1. **제목**: 모멘츠(친구 목록)에서 클릭을 유도할 수 있는가? 잘릴 위험은 없는가(30자 초과)?
2. **도입부**: 첫 3문단이 독자를 붙잡는가? 훅이 충분히 강한가?
3. **구조**: 논리 흐름이 매끄러운가? 비약이나 단절은 없는가?
4. **깊이와 가독성의 균형**: 수식/기술 부분이 실제로 쉽게 이해되는가? "쉬운 척했지만 제대로 설명하지 않은" 부분은 없는가?
5. **리듬**: 지나치게 긴 문단은 없는가? 각 절의 길이는 적절한가?
6. **삽화**: 이미지가 실제로 삽입되어 있는가(자리표시자가 아님)? 독자가 시각적 도움이 가장 필요한 지점에 위치하는가?
7. **마무리**: 전파력이 있는가? 다 읽은 독자가 공유하고 싶어질까?

## 아티클 전문
{완성된 초고}

## 출력 형식
1. 총평(3~5문장)
2. 제목 수정 제안(2~3개 후보 제시)
3. 절별 수정 제안("원문 → 수정 제안"의 구체적 대조 형태로 제시)
4. 가장 중요한 개선 포인트 3가지
```

### 독자 에이전트 프롬프트 템플릿

```
당신은 {타깃 독자 프로필}(Reader Agent)이다. 다음 아티클을 독자의 관점에서 검토하라.

## 당신의 배경
{타깃 독자의 지식 수준과 독서 습관에 대한 구체적 설명}

## 아티클 전문
{완성된 초고}

## 다음 질문에 답하라
1. 첫 3문단을 읽고 계속 읽고 싶어지는가? 그 이유는?
2. 어느 부분이 "이해가 안 되거나" "다시 읽어야 이해되는가"? 구체적으로 어느 문장인가?
3. 기술/수식 부분을 이해했는가? "쉬운 말 번역"이 도움이 되었는가?
4. 아티클의 핵심 비유는 적절한가? 더 나은 비유가 있는가?
5. 너무 길거나 짧은가? 어디서 인내심을 잃을 것 같은가?
6. 다 읽은 후 아티클의 핵심 논지를 한 문장으로 요약할 수 있는가?
7. 이 아티클을 공유하겠는가? 공유한다면 어떤 말을 덧붙이겠는가?
8. 알고 싶었지만 아티클이 다루지 않은 질문이 있는가?
```

## 4단계: 최종 원고 확정

### 4-1: 두 에이전트의 피드백 종합

다음의 빈번한 문제 유형에 집중한다:

| 문제 유형 | 편집자의 흔한 피드백 | 독자의 흔한 피드백 | 처리 방식 |
|---------|------------|------------|---------|
| 도입부가 약함 | 훅이 부족함 | 첫 3문단에서 계속 읽을 동력이 없음 | 데이터/반직관적 결론으로 도입부를 다시 작성 |
| 기술 단락에서 이탈 유발 | 수식이 너무 밀집됨 | 특정 단락을 3번 다시 읽어야 함 | 수식을 삭제하거나 이미지화하고, 더 직관적인 비유를 추가 |
| 리듬이 늘어짐 | 특정 절이 너무 김 | 특정 지점에서 인내심을 잃음 | 병합 또는 축약(특히 후반부에서 반복되는 기술 설명) |
| 마무리에 힘이 없음 | 전파력 부족 | 공유하지 않을 것 같음 | 캡처해서 공유할 수 있는 한 문장으로 다시 작성 |
| 개념의 비약 | 논리에 단절이 있음 | 특정 지점에서 "갑자기 이해가 안 됨" | 전환 문장이나 배경 설명 보완 |

### 4-2: 수정 실행

피드백에 따라 아티클을 다시 작성한다. 핵심 수정 원칙:

1. **편집자와 독자가 모두 지적한 문제는 반드시 수정한다**
2. **편집자만 지적한 문제는 높은 확률로 수정해야 한다**(편집자의 전문적 판단은 대체로 정확함)
3. **독자만 지적한 문제는 상황에 따라 수정한다**(독자 피드백은 실제 경험을 대변하지만, 모든 항목에 반드시 대응할 필요는 없음)
4. **양측이 상충할 때는 독자 쪽을 우선한다**(편집자는 완벽을 추구하지만, 독자 경험이 최종 기준이다)

### 4-3: 삽화 추출

논문 해설류 아티클은 최종 원고 확정 전에 반드시 삽화 추출을 완료해야 한다:

1. **렌더링**: `pdftoppm -png -r 900 -f {페이지번호} -l {페이지번호} 논문.pdf /tmp/page`(900 DPI부터 시작하며, 이미지가 500KB에 못 미치면 1200 또는 1500 DPI로 올린다)
2. **위치 확인**: 먼저 150 DPI로 전체 페이지를 렌더링하여 각 도표의 픽셀 좌표를 육안으로 확인한다
3. **자르기**: PIL로 좌표에 따라 잘라내고, `compress_level=1`로 저장하여 각 이미지가 500KB 이상이 되도록 한다
4. **저장**: `assets/{주제 약칭}/` 디렉토리에 저장하며, `fig{번호}-{설명}.png` 형식으로 이름을 붙인다
5. **삽입**: 아티클 안에서 `![설명](../../assets/{주제 약칭}/fig{번호}-{설명}.png)` 형식으로 참조한다

### 4-4: 최종 파일 산출

최종 원고를 md 파일로 저장하며, 파일 끝에 논문/자료의 원문 링크를 첨부한다:

```markdown
**논문 원문:**
- arXiv: {링크}
```

## 작문 레드라인

1. **데이터를 지어내지 않는다.** 인용하는 데이터는 반드시 출처가 있어야 하며, 찾을 수 없으면 "추정치"라고 표시한다
2. **AI스러운 어투를 쓰지 않는다.** "함께 살펴봅시다", "주목할 점은", "말하지 않을 수 없는 것은" 같은 상투적 표현을 금지한다
3. **과도하게 단언하지 않는다.** 기술 아티클에서 "파괴적인", "혁명적인" 같은 표현을 쓰지 않고, 데이터로 말한다
4. **수식에는 반드시 쉬운 말 설명을 붙인다.** 각 수식 뒤에는 "사람 말로 번역하면……"이라는 문단이 있어야 한다
5. **수식은 반드시 LaTeX로 작성한다.** `$$...$$` 형식을 사용하며, 순수 텍스트 수식을 금지한다
6. **삽화는 반드시 실제로 삽입한다.** 논문 해설류는 PDF에서 고화질 원본 그림(500KB 이상)을 추출하며, `[그림X]` 같은 자리표시자를 금지한다
7. **표의 괄호 설명은 정확해야 한다.** 개념을 설명할 때는 정확한 정의를 사용하고, 모호한 동사구를 쓰지 않는다(예: "교사가 만든 텍스트"라고 해야지 "교사를 배우는 텍스트"라고 하지 않는다)
8. **비유는 일관되게 유지한다.** 전체 아티클에서 하나의 주된 비유를 사용하며, 절마다 새로운 비유로 바꾸지 않는다
9. **마무리는 반드시 전파력이 있어야 한다.** 마지막 문장은 그 자체로 캡처해서 공유할 만한 가치가 있어야 한다
